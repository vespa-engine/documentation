We recommend configuring node-local storage for the content cluster (e.g. the search core) to maximize performance by avoiding network I/O on the data path. In a standard Vespa deployment, this is controlled through the storage-type attribute under the resources tag in the application package. However, that attribute has no effect when running Vespa on Kubernetes. Instead, local storage should be configured through the storageClass field in the VespaSet resource. Vespa on Kubernetes abstracts away the concept of storage entirely, and will consume whatever is provided by the referenced storage class.

To provision a local storage class, we recommend using the Local Persistent Volume static provisioner provided by the Kubernetes SIGs, which will manage the lifecycle of individual disks on your nodes and expose a standard volume API to consumers. This eliminates the need to manually configure node affinity rules on each Pod for data locality, as the Kubernetes scheduler is aware of the node associated with each Local Persistent Volume and ensures that the attached Pod is scheduled onto the correct node.

Install the Local Volume Provisioner

Clone the provisioner repository and install it onto the cluster using Helm: 

$ git clone git@github.com:kubernetes-sigs/sig-storage-local-static-provisioner.git
$ cd sig-storage-local-static-provisioner
$ helm install -f helm/examples/baremetal-default-storage.yaml local-volume-provisioner \
    --namespace kube-system ./helm/provisioner

Verify the Storage Class

After installation, confirm that the local-storage storage class is present: 

$ kubectl get storageclasses
NAME                      PROVISIONER                    RECLAIMPOLICY   VOLUMEBINDINGMODE      ALLOWVOLUMEEXPANSION   AGE
local-storage (default)   kubernetes.io/no-provisioner   Delete          WaitForFirstConsumer   false                  12h
standard (default)        k8s.io/minikube-hostpath       Delete          Immediate              false                  14h

$ kubectl get storageclass local-storage
NAME                      PROVISIONER                    RECLAIMPOLICY   VOLUMEBINDINGMODE      ALLOWVOLUMEEXPANSION   AGE
local-storage (default)   kubernetes.io/no-provisioner   Delete          WaitForFirstConsumer   false                  12h

Ensure the storage class has a VolumeBindingMode of WaitForFirstConsumer. This enables volume topology-aware scheduling, so the Kubernetes scheduler can place each Pod on the node that owns its Persistent Volume before binding occurs.

Use the Storage Class in a VespaSet

Set storageClass to local-storage for both the configServer and application sections of the VespaSet resource: 

$ cat > vespaset.yaml <<EOF
apiVersion: k8s.ai.vespa/v1
kind: VespaSet
metadata:
  name: vespaset-sample
  namespace: ${NAMESPACE}
spec:
  version: "${VESPA_VERSION}"

  configServer:
    image: "${VESPA_OPERATOR_IMAGE}"
    storageClass: "local-storage"
    generateRbac: false

  application:
    image: "${VESPA_IMAGE}"
    storageClass: "local-storage"

  ingress:
    endpointType: "NONE"
EOF

$ kubectl apply -f vespaset.yaml

Other Local Storage

Refer to the Installation guide for the full set of VespaSet fields and how to deploy a Vespa application once the cluster is running.

Users of this mode should be cautious when performing resource changes through the application package. If a Pod's resources are set to a value that exceeds the Kubernetes node's remaining capacity, the Pod will become unschedulable, since it is now sticky to that node due to data colocation. This may also occur during an upgrade in resource-contention scenarios, where Pods from external namespaces are competing for capacity on the same node. In these cases, we recommend exclusively reserving the content Pod's node through nodeAffinity rules set in a PodTemplate.
