---
title: "Partial Updates"
---

A partial update is an update to one or more fields in a document. It also includes updating all index structures so the effect of the partial update is immediately observable in queries. See [document update JSON format](/en/reference/schemas/document-json-format#update) for full details on the various possible partial update operations.

<Frame>
![Attribute is an in-memory data structure](/assets/img/attributes-update.svg)
</Frame>

In Vespa, all fields can be partially updated by default. A field is index, attribute or summary or a combination of these, and both index and attribute fields can be queried.

- For [index](/en/content/proton#index) and summary fields, an update means a read-modify-write to the [document store](/en/content/proton#document-store) and limits throughput.
- Most [attribute](/en/content/attributes) fields do not require the document store read-modify-write, increasing write throughput by orders of magnitude. The following attribute types require a read-modify-write to the document store:
  - [array of struct](/en/reference/schemas/schemas#array)
  - [map of primitive and struct](/en/reference/schemas/schemas#map)
  - [predicate](/en/reference/schemas/schemas#predicate)
  - [reference](/en/reference/schemas/schemas#reference)

<Warning>
**Important:**

For highest possible write throughput for field updates, use attributes to write at memory speed.
</Warning>

## Use cases

Partial updates have many use cases. *Functionally*, it enables updating a document without anything else than the ID, simplifying logic in the upper levels of the serving stack. *Performance-wise*, partial updates enables applications with a real-time update flow in tens of thousands updates per second.       <td>For all <a href="/en/reference/schemas/schemas#index" aria-label="indexed fields">indexed fields</a>, a memory index is used for the recent changes, implemented using B-trees. This is periodically <a href="/en/content/proton#memory-index-flush" aria-label="flushed">flushed</a> to a disk-based posting list index. Disk-based indexes are subsequently <a href="/en/content/proton#disk-index-fusion" aria-label="merged">merged</a>. <br/><br/> Updating the in-memory B-trees is lock-free, implemented using copy-on-write semantics. This gives high performance, with a predictable steady-state CPU/memory use. The driver for this design is the requirement for a sustained, high change rate, with stable, predictable read latencies and small temporary increases in CPU/memory. This compared to index hierarchies, merging smaller real-time indices into larger, causing temporary hot-spots.<br/><br/>   When updating an indexed field, the document is read from the <a href="/en/content/proton#document-store" aria-label="document store">document store</a>, the field is updated, and the full document is written back to the store. At this point, the change is searchable, and an ACK is returned to the client. Use <a href="/en/content/attributes/" aria-label="attributes">attributes</a> to avoid such document disk accesses and increase performance for partial updates. Find more details in <a href="/en/performance/sizing-feeding/" aria-label="feed performance">feed performance</a>.</td>

Note that operations are persisted to the Transaction Log Service (TLS), in the rare case of a power failure or unclean shutdown, the operations are synced from the TLS.<br/><br/>   Note there is no transactional support for updates across fields. This is a throughput vs consistency tradeoff that enables the extreme update rates without being a practical limitation for many applications. More details in <a href="/en/content/attributes/" aria-label="attributes">attributes</a>.<br/><br/>  Updating <a href="/en/querying/searching-multivalue-fields" aria-label="multivalue">multivalue</a> attributes (arrays, maps, sets, tensors) means reading the current value, making the update and writing it back:<br/><br/>  • <a href="/en/reference/schemas/schemas#array" aria-label="Array of primitive types">Array of primitive types</a>, <a href="/en/reference/schemas/schemas#weightedset" aria-label="weightedsets">weightedsets</a> and <a href="/en/reference/schemas/schemas#tensor" aria-label="tensors">tensors</a> are in memory and therefore fast, see <a href="/en/content/attributes/#data-structures" aria-label="attribute data structures">attribute data structures</a> for performance considerations. <br/><br/> • If the attribute field is an <a href="/en/reference/schemas/schemas#array" aria-label="array of struct">array of struct</a> or <a href="/en/reference/schemas/schemas#map" aria-label="map">map</a>, values are written in the document store and update rates are hence lower - refer to <a href="https://github.com/vespa-engine/vespa/issues/10892" aria-label="#10892 updates of array of map/struct">#10892 updates of array of map/struct</a>. <br/><br/> Query execution time can be improved by adding an in-memory B-tree posting list structure using <a href="/en/performance/feature-tuning#when-to-use-fast-search-for-attribute-fields" aria-label="fast-search">fast-search</a>. This increases work when updating, as both the value and the posting list is updated and hence decreases update throughput.<br/><br/>  See <a href="/en/performance/sizing-feeding/#attribute-store" aria-label="sizing-feeding">sizing-feeding</a> for how to ensure an attribute is in memory on all nodes with a replica (searchable-copies or fast-access).</td>
