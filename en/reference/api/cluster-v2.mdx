<table className="vespa-table">
  <thead>
    <tr>
      <th>State</th>
      <th>Description</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <td><code>{`Up`}</code></td>
      <td>The node is up and available to keep buckets and serve requests.</td>
    </tr>
    <tr>
      <td><code>{`Down`}</code></td>
      <td>The node is not available, and can not be used.</td>
    </tr>
    <tr>
      <td><code>{`Stopping`}</code></td>
      <td>This node is stopping and is expected to be down soon. This state is typically only exposed to the cluster controller to tell why the node stopped. The cluster controller will expose the node as down or in maintenance mode for the rest of the cluster. This state is thus not seen by the distribution algorithm.</td>
    </tr>
    <tr>
      <td><code>{`Maintenance`}</code></td>
      <td>This node is temporarily unavailable. The node is available for bucket placement, so redundancy is lower. Using this mode, new replicas of the documents stored on this node will not be created, allowing the node to be down with less of a performance impact on the rest of the cluster. This mode is typically used to mask a down state during controlled node restarts, or by an administrator that need to do some short maintenance work, like upgrading software or restart the node.</td>
    </tr>
    <tr>
      <td><code>{`Retired`}</code></td>
      <td>A retired node is available and serves requests. This state is used to remove nodes while keeping redundancy. Buckets are moved to other nodes (with low priority), until empty. Special considerations apply when using <a href="/en/content/elasticity#grouped-distribution" aria-label="grouped distribution">grouped distribution</a> as buckets are not necessarily removed.</td>
    </tr>
  </tbody>
</table>

      <td>&#42;<code>{`<identifier>`}</code>&#42; (. &#42;<code>{`<identifier>`}</code>&#42; )&#42;</td>
      <td>The hierarchical group assignment of a given content node. This is a dot separated list of identifiers given in the application services.xml configuration.</td>
    </tr>
    <tr>
      <td>node</td>
      <td>[0-9]+</td>
      <td>The index or distribution key identifying a given node within the context of a content cluster and a service type.</td>
    </tr>
    <tr>
      <td>service-type</td>
      <td>(distributor|storage)</td>
      <td>The type of the service to look at state for, within the context of a given content cluster.</td>
    </tr>
    <tr>
      <td>state-disk</td>
      <td>(up|down)</td>
      <td>One of the valid disk states.</td>
    </tr>
    <tr>
      <td>state-unit</td>
      <td><a href="#up" aria-label="up">up</a> | <a href="#stopping" aria-label="stopping">stopping</a> | <a href="#down" aria-label="down">down</a></td>
      <td>The cluster controller fetches states from all nodes, called &#42;unit states&#42;. States reported from the nodes are either <code>{`up`}</code> or <code>{`stopping`}</code>. If the node can not be reached, a <code>{`down`}</code> state is assumed. This means, the cluster controller detects failed nodes. The subsequent &#42;generated states&#42; will have nodes in <code>{`down`}</code>, and the <a href="/en/content/idealstate" aria-label="ideal state algorithm">ideal state algorithm</a> will redistribute <a href="/en/content/buckets" aria-label="buckets">buckets</a> of documents.</td>
    </tr>
    <tr>
      <td>state-user</td>
      <td><a href="#up" aria-label="up">up</a> | <a href="#down" aria-label="down">down</a> | <a href="#maintenance" aria-label="maintenance">maintenance</a> | <a href="#retired" aria-label="retired">retired</a></td>
      <td>Use tools for <a href="/en/operations/self-managed/admin-procedures#cluster-state" aria-label="user state management">user state management</a>. Retire a node from a cluster - use <code>{`retired`}</code> to move buckets to other nodes Short-lived maintenance work - use <code>{`maintenance`}</code> to avoid merging buckets to other nodes Fail a bad node. The cluster controller or an operator can set a node <code>{`down`}</code></td>
    </tr>
    <tr>
      <td>state-generated</td>
      <td><a href="#up" aria-label="up">up</a> | <a href="#down" aria-label="down">down</a> | <a href="#maintenance" aria-label="maintenance">maintenance</a> | <a href="#retired" aria-label="retired">retired</a></td>
      <td>The cluster controller generates the cluster state from the <code>{`unit`}</code> and <code>{`user`}</code> states, over time. The generated state is called the &#42;cluster state&#42;.</td>
    </tr>
